# How to Migrate Between Deployments

## 适用范围
- 适用于把一个部署中的业务数据迁移到另一个新部署
- 适用于 Docker、自托管 Linux、iStoreOS、macOS、Windows Docker Desktop

## 结论
- 对当前版本来说，标准迁移方式就是：旧部署导出备份 → 新部署导入备份
- 但这不等于“什么都不用管”，部署级环境变量仍需手动配置

## 迁移前提
- 旧部署与新部署都使用包含 `schemaVersion = 2` 备份能力的版本
- 新部署已经正常启动，并完成数据库迁移
- 若新部署使用 Docker，确保 `/app/data` 已正确持久化

## 迁移步骤

### 1. 在旧部署导出备份
- 进入 `设置` 页面
- 在“备份与恢复”区域点击“下载备份 (.json)”
- 该备份会包含：
  - Todo / 子任务
  - 纪念日
  - 订阅
  - 物品
  - 通知发送记录
  - 周报/月报发送记录
  - 通知渠道配置
  - 系统内定时任务配置
  - 图标缓存与品牌元数据

### 2. 准备新部署
- 先部署并启动新实例
- 确认应用版本不早于导出端版本
- 若依赖环境变量，先补齐部署配置

### 3. 在新部署导入备份
- 打开新部署的 `设置` 页面
- 在“备份与恢复”区域选择刚导出的 `.json`
- 迁移整套数据时，使用“覆盖导入”
- 仅想把部分数据并入现有实例时，使用“合并导入”

### 4. 导入后验证
- 检查首页、Todo、纪念日、订阅、物品页面
- 检查通知渠道配置是否已恢复
- 检查系统内定时任务开关与时间设置是否正确
- 建议手动发送一次测试通知

## 覆盖导入 vs 合并导入

### 覆盖导入
- 适合“把旧部署完整搬到新部署”
- 会恢复完整系统设置
- 会恢复通知与摘要发送记录
- 是跨机器迁移的推荐方式

### 合并导入
- 适合把一份备份追加到当前已有数据中
- 不会覆盖当前部署的系统设置
- 相同主键的数据会跳过

## 不会自动迁移的内容
- `NOTIFY_CRON_SECRET`
- `DATABASE_FILE_PATH`
- Docker 端口映射、Volume 挂载、反向代理配置
- 任何部署平台自身配置

## 推荐做法
- 迁移整套实例时，优先使用“覆盖导入”
- 导入完成后，手动补齐环境变量，再做一次通知测试
- 若目标是 100% 复制整个运行环境，除导入备份外，还要同步部署配置
